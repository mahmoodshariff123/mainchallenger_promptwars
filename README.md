# MonsoonReady — GenAI-powered Monsoon Preparedness & Citizen Assistance

A working web app that helps **families, facilities, and communities** prepare for
the monsoon. Every dynamic feature is generated by a **real generative-AI call,
grounded in real weather data** — there are no hardcoded pages, no mock data, and
the app never fabricates weather or alerts.

## What it does
| Feature | How it's powered (all real) |
|---|---|
| Live conditions & real-time alerts (before/during/after) | **Open-Meteo** live observations + 7-day forecast; severity computed deterministically from real readings, then summarized by the model |
| Personalized preparedness plan | Real forecast + your household profile → model generates before/during/after actions |
| Emergency checklist | Model builds a kit/list adapted to your household & real rainfall/wind risk |
| Travel advisory | Origin + destination real forecasts → model risk assessment |
| Multilingual assistant | Any language (English, Hindi, Marathi, Tamil, Spanish, French, Arabic, Urdu…) |
| Household profiles & auth | Session auth (signed httpOnly cookies, PBKDF2-hashed passwords) |

**Anti-hallucination design:** the model only ever receives the *actual* weather
JSON and household facts, and is instructed to use nothing else. Alert severity is
computed from documented meteorological thresholds (not by the model), so there are
**no false positives** and **no invented data**. If the model/key is unavailable the
endpoint returns an honest `503` — never fake content.

## Quick start (local)
```bash
pip install -r requirements.txt
# Provide a GenAI key (any OpenAI-compatible provider):
#   set OPENAI_API_KEY=sk-...          # OpenAI
#   or OPENAI_BASE_URL + key for OpenRouter / Together / Groq / a local Ollama
python run.py
# open http://localhost:8000
```
Test credentials (auto-created on first run):
**`demo@monsoonready.app` / `Monsoon@123`**

Without a key the app still runs: live weather + deterministic alerts work fully,
and AI features return a clear "GenAI unavailable" message instead of fabricating.

### Run the GenAI features with NO external API (local model)
```bash
ollama serve
ollama pull llama3
# then in .env:
# OPENAI_API_KEY=ollama
# OPENAI_BASE_URL=http://localhost:11434/v1
# OPENAI_MODEL=llama3
```

## Tests (end-to-end)
```bash
python -m pytest tests/test_e2e.py -q
```
Covers health, register/login/me/logout, demo login, real weather, real+honest
alerts, profile save/get, auth-gating of AI features, and honest failure when no
key is set.

## Deploy to Netlify
This is a Python backend, so it runs on Netlify as a **Python Function** (Mangum
adapter). `netlify/functions/api.py` wraps the FastAPI app.
```bash
netlify login
netlify deploy --prod --dir static
```
Set in the Netlify UI (Site settings → Environment):
`OPENAI_API_KEY`, `OPENAI_BASE_URL` (optional), `OPENAI_MODEL`, `SESSION_SECRET`.
`DB_PATH` and `PYTHON_VERSION` are already set in `netlify.toml` (SQLite uses
`/tmp` on the read-only function filesystem). The `[[redirects]]` rules send `/api/*`
to the function and serve the SPA for everything else.

> Note: on serverless, the SQLite DB is per-instance/ephemeral — fine for a demo.
> For production use a managed DB (Postgres) and point `DB_PATH`/swap `db.py`.
